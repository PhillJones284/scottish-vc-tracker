CONTEXT
=======

This is the scottish-vc-tracker project. It is a pipeline that monitors news sources for
Scottish VC investment activity and produces a weekly intelligence report.

The pipeline has four stages:
  Stage 1 — Scraper    (Claude agent: .claude/agents/scraper.md)
  Stage 2 — Parser     (Claude agent: .claude/agents/parser.md)
  Stage 3 — Deduplicator (Claude agent: .claude/agents/deduplicator.md)
  Stage 4 — Reporter   (Claude agent: .claude/agents/reporter.md)

Currently all four stages run as Claude Code subagents, invoked by the orchestrator
instructions in CLAUDE.md. The goal of this task is to convert Stages 2 and 3 to
deterministic Python code, keeping Stages 1 and 4 as Claude agents.

This is a fresh copy of the repo. Do not modify any data files or config files unless
explicitly instructed. Do not run the pipeline.

STAGE 2 — PARSER (pipeline/parser.py)
======================================

Input:  All files matching data/raw/YYYY-MM-DD_*.json (ignore errors.json)
Output: data/processed/investments.json

The parser must implement exactly the logic described in .claude/agents/parser.md.
Read that file for the full specification. Key points:

1. AGGREGATE
   - Load all raw files, combine into one list of records

2. NORMALISE each record:

   Company name:
   - Title case unless the company stylises differently
   - Strip legal suffixes (Ltd, Limited, plc) for matching but preserve in display name
   - Flag ambiguous/generic names

   Location:
   - Map to one of: Edinburgh | Glasgow | Aberdeen | Dundee | Inverness | St Andrews |
     Other Scotland | Unknown
   - "Scottish" without a city → Other Scotland

   Sector:
   - Load config/sectors.json, normalise to a listed category
   - If no match: keep raw value, set sector_normalised: false

   Round type:
   - Normalise to: Pre-Seed | Seed | Series A | Series B | Series C+ | Growth | Bridge | Unknown

   Amount:
   - Convert to GBP millions using rates in config/fx_rates.json
   - Store as amount_gbp_millions (float or null)
   - Preserve original string in amount_original
   - Parse strings like "£4.2m", "$10m", "€5 million", "undisclosed"

   Investors:
   - Combine lead_investor + other_investors, deduplicate
   - Normalise VC names using config/known_vcs.json (check canonical_name and aliases)

   Date:
   - Parse to ISO 8601 (YYYY-MM-DD) or null
   - Attempt to extract from URL or headline if announcement_date is null

3. SCORE each record (data_quality_score, 0–100):
   - Company name present: +20
   - Location identified: +10
   - Sector identified: +10
   - Round type known: +10
   - Amount known: +15
   - At least one investor named: +20
   - Date known: +10
   - Source is primary (press release / company blog): +5

4. GENERATE RECORD ID:
   Format: {normalised-company-name}_{round-type}_{announcement-date}
   - normalised-company-name: lowercase, spaces→hyphens, strip legal suffixes, keep only
     letters/digits/hyphens, collapse consecutive hyphens
   - round-type: lowercase, spaces→hyphens (e.g. seed, series-a, series-c+, unknown)
   - announcement-date: YYYY-MM-DD, or "undated" if null
   Examples: wallet-ai_series-a_2026-03-15 | medtech-scotland_seed_undated

5. FLAG issues array:
   - "amount_missing"
   - "investor_unnamed"
   - "date_missing"
   - "location_unknown"
   - "possible_grant_not_vc"
   - "company_not_clearly_scottish"

6. OUTPUT schema (write to data/processed/investments.json):
   {
     "generated_at": "ISO8601 timestamp",
     "record_count": 42,
     "source_files": ["list of raw files consumed"],
     "parse_errors": ["list of filenames that failed to parse"],
     "investments": [
       {
         "id": "string",
         "company_name": "string",
         "company_location": "Edinburgh | Glasgow | ...",
         "company_sector": "string",
         "sector_normalised": true,
         "round_type": "string",
         "amount_original": "string or null",
         "amount_gbp_millions": 4.2,
         "currency_original": "GBP",
         "investors": ["Investor A", "Investor B"],
         "lead_investor": "string or null",
         "announcement_date": "YYYY-MM-DD or null",
         "source_url": "string",
         "source_name": "string",
         "headline": "string",
         "summary": "string",
         "confidence": "high | medium | low",
         "data_quality_score": 85,
         "issues": [],
         "raw_snippet": "string"
       }
     ]
   }

Preserve all records regardless of quality. Never discard.


STAGE 3 — DEDUPLICATOR (pipeline/deduplicator.py)
===================================================

Input:  data/processed/investments.json
        data/processed/ledger.json (may not exist on first run)
Output: data/processed/investments_deduped.json
        data/processed/ledger.json (updated in place)

The deduplicator must implement exactly the logic described in .claude/agents/deduplicator.md.
Read that file for the full specification. Key points:

1. MATCHING RULES (use rapidfuzz for fuzzy string comparison):

   Definite match (merge automatically):
   - Same company name (fuzzy score >90%) AND same round type AND dates within 60 days
   - Same company name AND same amount AND same investors

   Probable match (merge with flag):
   - Same company name AND same round type, but one or both records lack a date
   - Same company name AND overlapping investor lists, dates within 90 days

   Possible match (flag for review, do NOT merge):
   - Same company name, different round type
   - Very similar company names (could be different companies)

2. MERGE STRATEGY:
   - Keep the record with the higher data_quality_score as base
   - Fill null fields from the lower-quality record
   - Combine source URLs into source_urls array
   - Add duplicate_of: null to canonical record
   - Add source_count: N
   - Add merge_confidence: "definite" | "probable"

3. LEDGER MANAGEMENT:
   - If ledger.json exists, load it; otherwise start with empty list
   - For each deduped record:
     - If it matches an existing ledger entry (by id) → update it (refresh last_seen, add sources)
     - If new → append to ledger
   - Add first_seen and last_seen (YYYY-MM-DD) to every ledger entry
   - Write updated ledger back to data/processed/ledger.json
   - IMPORTANT: treat ledger.json with care — it is the persistent historical record

4. OUTPUT schema (write to data/processed/investments_deduped.json):
   {
     "generated_at": "ISO8601",
     "run_date": "YYYY-MM-DD",
     "stats": {
       "input_records": 47,
       "after_dedup": 31,
       "new_this_run": 18,
       "updated_existing": 13,
       "flagged_for_review": 2
     },
     "investments": [
       {
         "...all fields from parser output...",
         "source_urls": ["url1", "url2"],
         "source_count": 2,
         "merge_confidence": "definite",
         "first_seen": "YYYY-MM-DD",
         "last_seen": "YYYY-MM-DD",
         "is_new_this_run": true
       }
     ],
     "flagged_for_review": [
       {
         "reason": "possible_duplicate",
         "records": ["id1", "id2"],
         "note": "human-readable explanation"
       }
     ]
   }

Be conservative: when in doubt, flag rather than merge.


ORCHESTRATOR (pipeline/run.py)
================================

The orchestrator replaces the pipeline instructions in CLAUDE.md. It should:

1. Accept an optional --date argument (default: today's date in YYYY-MM-DD format)

2. Run Stage 1 (Scraper) as a Claude agent:
   - Read .claude/agents/scraper.md
   - Prepend "Today's date is YYYY-MM-DD." with the actual date
   - Invoke Claude to run the agent (see CLAUDE AGENT INVOCATION below)
   - Gate check: at least one data/raw/YYYY-MM-DD_*.json file exists with at least one record
   - If gate fails: print error message and exit with code 1

3. Run Stage 2 (Parser) by calling pipeline/parser.py directly as a Python function:
   - Call parser.run(date=date_str)
   - Gate check: data/processed/investments.json exists and record_count > 0
   - If gate fails: print error and exit 1

4. Run Stage 3 (Deduplicator) by calling pipeline/deduplicator.py directly:
   - Call deduplicator.run(date=date_str)
   - Gate check: data/processed/investments_deduped.json exists
   - If gate fails: print error and exit 1

5. Run Stage 4 (Reporter) as a Claude agent:
   - Read .claude/agents/reporter.md
   - Prepend today's date
   - Invoke Claude to run the agent
   - Gate check: data/reports/YYYY-MM-DD_vc-report.md exists
   - If gate fails: print error and exit 1

6. On success: print "Pipeline complete. Report: data/reports/YYYY-MM-DD_vc-report.md"

CLAUDE AGENT INVOCATION:
Use the Anthropic Python SDK to invoke Claude for stages 1 and 4. Use claude-sonnet-4-6
as the model. The agent prompts (scraper, reporter) need Read, Write, WebFetch, WebSearch,
and Bash tools — use Claude's built-in tool_use with appropriate tool definitions, or
alternatively use the claude CLI subprocess if simpler. Whichever approach you choose,
explain the trade-offs in a comment in run.py.

The ANTHROPIC_API_KEY environment variable must be set. If it is not, print a clear error
and exit before attempting any API calls.
